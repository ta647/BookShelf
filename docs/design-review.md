## 1. 画面遷移の設計

| 操作 | 正常系リダイレクト先 | 異常系（バリデーションエラー） |
|------|---------------------|-------------------------------|
| 書籍登録成功 | /books（一覧） | 入力内容を保持したまま /books/create に戻る |
| 書籍更新成功 | /books/{book}（詳細） | 入力内容を保持したまま /books/{book}/edit に戻る |
| 書籍削除成功 | /books（一覧） | - |
| ジャンル登録成功 | /genres（一覧） | 入力内容を保持したまま /genres/create に戻る |

| ジャンル削除成功 | /genres（一覧） | - |
| レビュー投稿成功 | /books/{book}（詳細） | 入力内容を保持したまま /books/{book} に戻る |
| レビュー更新成功 | /books/{book}（詳細） | 入力内容を保持したまま /reviews/{review}/edit に戻る |
| レビュー削除成功 | /books/{book}（詳細） | - |
| ログイン成功 | /books（一覧） | 入力内容を保持したまま /login に戻る |

## 3. 認可ルールの設計

| リソース | 操作 | 権限 | Policy |
|---------|------|------|--------|
| 書籍 | 編集・更新 | 作成者本人のみ | BookPolicy@update |
| 書籍 | 削除 | 作成者本人のみ | BookPolicy@delete |
| レビュー | 編集・更新 | 投稿者本人のみ | ReviewPolicy@update |
| レビュー | 削除 | 投稿者本人のみ | ReviewPolicy@delete |
| ジャンル | 全操作 | 認証済みユーザー全員 | Policy なし |
| お気に入り | 全操作 | 認証済みユーザー全員 | Policy なし |
| いいね | 全操作 | 認証済みユーザー全員 | Policy なし |

**認可エラー時の挙動:**
- 他人の書籍を編集しようとした場合 → 403 Forbidden
- 紐付く書籍があるジャンルを削除しようとした場合 → 削除を拒否してエラーメッセージを表示

**確認事項:**
- ジャンルの全操作を「認証済みユーザー全員」にした設計で問題ないですか？（管理者限定にすべきですか？）
- 認可エラー時は 403 を返す設計で合っていますか？

## 4. バリデーションルールの設計

### 書籍登録・編集

| フィールド | ルール |
|-----------|--------|
| title | 必須・文字列・最大255文字 |
| author | 必須・文字列・最大255文字 |
| isbn | 必須・13桁の数字・一意（編集時は自身を除外） |
| published_date | 必須・有効な日付 |
| description | 任意・文字列 |
| image_url | 任意・URL形式 |
| genres | 必須・配列・1つ以上 |
| genres.* | 整数・genres テーブルに存在する |

### レビュー投稿・編集

| フィールド | ルール |
|-----------|--------|
| rating | 必須・整数・1〜5 |
| comment | 任意・文字列・最大1000文字 |

### ジャンル登録・編集

| フィールド | ルール |
|-----------|--------|
| name | 必須・文字列・最大255文字・一意（編集時は自身を除外） |

### 公開API 書籍一覧（検索パラメータ）

| パラメータ | ルール |
|-----------|--------|
| keyword | 任意・文字列・最大255文字 |
| genre | 任意・整数・genres テーブルに存在する |
| page | 任意・整数・1以上 |
| per_page | 任意・整数・1〜100 |

### 公開API 書籍登録

Web版と同等のルールに加えて：

| フィールド | ルール |
|-----------|--------|
| user_id | 必須・整数・users テーブルに存在する |

**確認事項:**
- ISBNの桁数チェック（13桁固定）はこれで合っていますか？
- レビューのコメントを最大1000文字にしましたが、この制約で問題ないですか？
